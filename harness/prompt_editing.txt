## Working approach (bank a floor, confirm the target, then edit)
You have a soft budget of about 15 minutes — enough for several refinement passes. Use it; do not stop at the first valid solid.

1. Import and BANK A FLOOR FIRST. import_cad_file("input.step", "part"), then immediately validate() -> export() -> save_snapshot() the solid UNCHANGED, before any edit. The unmodified input is itself a valid, scored fallback (the editing benchmark scores a no-op against a baseline), so this guarantees a non-zero result on disk no matter what happens next. Do this as your very first actions — do NOT go feature-hunting first. If the unchanged import does not PASS validate()/the export gate, the input is itself defective — attempt minimal healing (e.g. ShapeFix) to bank a valid floor before editing; do not start the edit on an unbankable solid.
2. Identify and CONFIRM the target features before changing anything. Enumerate the candidate features with execute() and report how many you found and where they are. Non-circular pockets are NOT holes — find_holes will not return them; inspect faces directly (by normal direction, axis orientation, and position relative to the bore centre). Check the set against the request (e.g. exactly "four pockets on the +X side, walls with long axis along Y") before you touch the geometry. If your count or locations do not match the request, keep looking — editing the wrong faces scores worse than the no-op floor. The request usually states the feature's CURRENT value (e.g. "from 20 mm to 30 mm"): measure your candidate's current value FIRST and confirm it matches that stated start — if it does not match, you have probably found the WRONG feature (keep searching), and if your candidate already reads the TARGET value, treat that as a red flag that you mis-identified the feature, not as a task already done. Honour every qualifier in the request (which side, which axis, "attached to two cylindrical sections", "long axis along Y").
   NOTE: import_cad_file registers "part" in the server's object registry but does NOT create a Python variable — referencing `part` in execute() raises NameError. In your first execute(), pull the solid into a namespace variable ONCE with `from build123d import import_step; part = import_step("input.step")`; the namespace persists across execute() calls, so edit that same `part` from then on (do NOT re-import each call — that discards your edits).
3. Apply ONLY the requested change to the imported solid, in place. Work from the confirmed features and the stated dimension (e.g. a 6 mm inward offset), not by eye. Do not rebuild the part from scratch; modify the existing B-rep so every untouched region stays exactly as imported. If the operation (fillet, cut, offset) FAILS on the feature the request names, fix the operation (smaller step, heal first, a different primitive) — do NOT silently retarget to a different feature just because the operation happens to succeed there; the described feature is the target even when it is harder to edit.
4. Re-checkpoint: validate(); only when it PASSES, export() and save_snapshot(), overwriting the floor with your edited result.
5. Iterate to fidelity (several passes): render_view() and compare against the reference renders — confirm the change is present and correct on EVERY target feature and that nothing else moved; use measure() to verify the edit dimension. Fix the largest discrepancy with execute(), then validate() again; only when it PASSES, export() and save_snapshot(). Repeat until the edit is correct and complete. Validity is NOT correctness: before finishing, prove the edit is the RIGHT one — re-measure the changed feature and compare to the banked floor; the changed dimension must have moved in the requested DIRECTION by the requested MAGNITUDE, on the feature the request names. The precise way to prove this is shape_compare: re-import the unedited input under a name — `show(import_step("input.step"), "original")` — then shape_compare("original", <your edited object>). It localises the change for you: confirm the changed region (changed.centroid / regions) sits on the feature the request names, that material moved the right way and amount (added_volume vs removed_volume; the surface displacement is max_deviation when magnitude_method=="exact_boolean", and for a cut/hole the exact magnitude is removed_volume/added_volume), and that unchanged_elsewhere is true (you disturbed nothing else). A "no change above the detection floor" result means the edit did NOT take and you must keep working — EXCEPT for a pure feature MOVE (sliding a hole sideways), which a surface compare cannot see: verify those by feature position with find_holes()/measure(). Banking the unchanged input is a safety floor, NOT an acceptable result: finishing with the input essentially unmodified — or with a valid solid that edited the wrong feature, wrong direction, or wrong amount — scores at or below the no-op floor. If you cannot confirm the output differs from the floor by the requested change, the edit did not take: keep working. Fall back to the unedited floor only if every valid edit attempt fails, and treat that as a failure, not completion.

## Rules (read carefully)
- Always leave a valid output.step on disk. The unchanged import counts as a valid result — bank it first (step 1) so you can never finish with nothing.
- Validity is judged by validate(), and authoritatively by the export() gate. measure() (dimensions, counts) and render_view() (appearance) do NOT reveal whether the solid is watertight and manifold — never infer validity from them or from a sensible-looking volume.
- After every geometry-changing execute(), call validate() before you render, measure, or export. Never build a new change on top of a solid you have not just re-validated.
- If a fix makes validate() FAIL, repairing it is the next action — or restore_snapshot() back to the last valid state — before anything else. Never keep iterating on an invalid solid.
- Never export a solid that did not just PASS validate(), so you never overwrite a good checkpoint with an invalid one.
- export() re-runs the validity gate with a STRICTER, exact mesh check than interactive validate() uses on a large solid — and an imported part being edited is large, so the exported result is the authoritative verdict. Always read export()'s output: a "VALIDITY GATE FAIL" there means the file is invalid even if validate() had PASSED; fix it and re-export.
- Edit in place; change only what the request asks. Do not re-model untouched regions, and do not add or remove features the request does not mention.
- Never probe geometry by sampling is_inside() over a grid of points — it is slow on large solids and risks the 120 s execution timeout, which destroys all session state. To inspect a cross-section use cross_sections() / render_view(clip_plane=...); to inspect individual features iterate .faces() and read .normal_at(), .center(), and .area().
- Render to COMPARE after a change, not to perceive each step: at most one render per view per fidelity pass, and never re-render a view without an intervening geometry change. Prefer measure() for exact dimensions.
- Use build123d operations (Part +/-, fillet, chamfer, offset) for geometry. Drop to raw OCP/OCCT only for repair you genuinely cannot express in build123d — raw OCCT booleans are slow on large solids and easy to get wrong.
